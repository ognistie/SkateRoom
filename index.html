01 — SOBRE O PROJETO

SKATEROOM

Plataforma de watch party inspirada pelo filme Mid 90s e pela cultura de skate underground — onde música, skate e rap underground se encontram numa garagem virtual de Compton.

📺 O que é

Uma sala virtual onde amigos entram, escolhem personagens pixel art, sentam no sofá e assistem vídeos de skate juntos — tudo sincronizado em tempo real via WebSocket.

🎯 O problema

Não existe um lugar com estética underground real onde a galera do skate e rap possa assistir vídeos juntos como se estivessem numa garagem. SkateRoom resolve isso.

02 — ARQUITETURA

Stack Técnico & Arquitetura de Arquivos

SkateRoom/
├── index.html — SPA principal (todas as páginas)
├── css/style.css — Estilos, layout, animações, responsivo
├── js/app.js — Lógica principal, YouTube API, Socket.io client
├── js/characters.js — 12 personagens pixel art em SVG
├── js/playlist.js — 22 vídeos de skate (YouTube IDs)
├── server/index.js — Backend Node.js + Socket.io
├── server/package.json — Dependências (express, socket.io, cors)
├── Dockerfile — Container para deploy
└── DEPLOY-AWS.md — Guia completo EC2

🌐 HTML5 + SVG

index.html · 57KB

Single Page Application — todas as páginas (Home, Sala, Skate Builder, Bonde, Sobre, News) em um único HTML. Cenário da sala desenhado em SVG inline puro: parede de tijolos via <pattern>, graffiti via <path>, neon com <filter> + <animate>. Personagens são SVG com image-rendering:pixelated.

Principais elementos: <svg viewBox="0 0 1000 600"> · <pattern> bricks · <filter> glow · <animate> neon · <foreignObject> overlay

🎨 CSS3

style.css · 17KB

CSS Grid para layout da sala (grid-template-columns:1fr 300px). CSS Variables para paleta (--blood-red, --amber, --purple). YouTube overlay posicionado com position:absolute em porcentagem sobre o SVG. VHS scanline via @keyframes vhsScan.

Lógica chave: .yt-overlay { left:36.4%; top:10.17%; width:27.2%; height:35.67% } — posiciona YouTube exatamente sobre a TV do SVG

⚡ JavaScript (Vanilla)

app.js · 11KB

SPA navigation via showPage(). YouTube IFrame API com onYouTubeIframeAPIReady. Flag isSyncing impede loop infinito: quando recebe sync do server, bloqueia onStateChange por 2 segundos. Personagens posicionados no sofá via createElementNS('http://www.w3.org/2000/svg','g').

Anti-loop: isSyncing=true → setTimeout(()=>{isSyncing=false},2000) — impede que onStateChange re-emita video-state pro server

📺 YouTube IFrame API

Externo via CDN

Player instanciado com new YT.Player('ytPlayer') usando playlist ID. Posicionado via CSS absoluto sobre a TV no SVG. Eventos: onReady, onStateChange (PLAYING/PAUSED), onError (skip automático). Métodos: loadVideoById(), seekTo(), getCurrentTime().

Sync: ytPlayer.loadVideoById({videoId, startSeconds}) — carrega mesmo vídeo na mesma posição pra quem entra na sala

🟢 Node.js + Express

server/index.js · 3KB

Servidor HTTP com express.static() servindo o frontend. API GET /api/health para monitoramento. Objeto rooms{} armazena estado: users[], videoId, position, isPlaying, lastUpdate. Cálculo de posição: pos += (Date.now() - lastUpdate) / 1000.

Comando: npm install express socket.io cors · node index.js · pm2 start index.js --name skateroom

🔄 Socket.io — WebSocket

Bidirecional real-time

Evento unificado video-state substitui play/pause/next separados. Usa socket.to(roomCode) (não io.to()) para enviar APENAS para outros — nunca de volta pro sender. No join-room, server calcula posição atual e envia pro novo user.

Eventos: join-room → room-update · video-state → sync-video · chat-message (broadcast) · disconnect → leaveRoom()

☁️ AWS EC2 + Docker

Deploy 24/7

EC2 Ubuntu 24.04 com PM2 (auto-restart, startup). Nginx como reverse proxy com suporte WebSocket (proxy_set_header Upgrade $http_upgrade). Docker disponível via Dockerfile multi-stage. Certbot para HTTPS.

Deploy: git clone → cd server && npm i → pm2 start index.js → nginx reverse proxy → certbot --nginx
